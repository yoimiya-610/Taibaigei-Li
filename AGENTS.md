# 李太白给协作指南

本文件约束在本仓库中协作的自动化代理与开发者。优先保持现有行为、数据兼容性和角色语气；不要为了局部改动进行无关重构。

## 硬规则

- 一个任务只对应一个 git 分支、一个 Codex 线程、一个 Pull Request。
- 一个线程只服务当前分支与当前 PR；新需求如果不属于同一目标，必须新开分支和新线程。
- 同一 PR 的实现、自测、PR 描述、Review 修复、CI 排查必须留在同一线程内完成。
- 保持现有行为、数据兼容性和角色语气；未经要求不要做无关重构。
- 不要提交 `.env`、`mybot/data/`、`mybot/logs/`、`backups/`、虚拟环境、数据库、本地适配器配置或公开发布副本目录。

## 新任务流程

1. 从最新 `main` 创建单一目标分支，例如 `fix/fortune-copy`、`feat/quote-search`。
2. 为该分支创建唯一对应的 Codex 线程。
3. 线程首条消息提供任务背景、目标、验收标准、约束和 PR 状态。
4. 先检查代码结构、影响模块、兼容性风险和验证方案，再开始改动。
5. 改动完成后，在当前分支提交并发起只聚焦单一目标的 PR。

线程标题建议格式：

`[Litaibaigei] [分支名] [PR号或主题]`

首条消息至少包含以下信息，允许等价表达：

```text
仓库路径：E:\Topdesk\PY\Litaibaigei
基线分支：main
工作分支：<feature_branch>
任务或 Issue：<链接或编号；没有则写“无”>
PR：<链接、编号或“待创建”>
目标：<本次要实现或修复的内容>

验收标准：
1. ...
2. ...
3. ...

约束：
- 保持现有行为、数据兼容性和李太白给角色语气
- 不做与本任务无关的重构

本线程只服务当前分支与当前 PR。请先检查代码结构、改动点、风险与验证方案，再开始实施。
```

## 项目结构

- `mybot/`：NoneBot2 + OneBot V11 机器人主体。
- `mybot/bot.py`：启动入口，同时加载 `plugins/` 与 `plugins_disabled/`。
- `mybot/common/`：共享工具层。新工具、存储、配置、AI 客户端应放在这里，不能放入 `plugins/`。
- `mybot/plugins/`：当前功能插件。
- `mybot/plugins_disabled/`：默认关闭、但可在运行时启用的五个历史小游戏实现，不是废弃代码。
- `mybot/data/`：运行数据，可能含 QQ 号、消息片段、积分、签到、称号和群配置。
- `mybot/logs/`、`backups/`：运行日志与备份，均属于私密运行产物。
- `tools/export_public_release.py`：导出不含私密运行信息的公开发布副本。
- `.github/pull_request_template.md`：PR 描述模板。
- `.github/workflows/ci.yml`：CI 定义。

本地运行：

```powershell
Copy-Item mybot/.env.example mybot/.env
uv sync --project mybot
uv run --project mybot li-taibaigei
```

也可在 `mybot/` 目录下执行：

```powershell
cd mybot
uv run python bot.py
```

## 角色与文案

机器人名为“李太白给”，角色为耄耋：唐装、墨镜、酒壶的李白式群聊角色，自称“本猫”。

- 所有面向用户的成功、失败、扣费、退款、确认、状态提示都应符合角色语境。
- 优先使用一小句动作旁白、描述或短诗；高频提示可以简短，但不能机械或暴露内部实现。
- 若收尾使用诗句，尽量以四句诗起步；仅在高频、极短提示场景下再酌情压缩。
- 不直接向用户说“API 调用失败”“模型调用失败”“函数错误”等技术语言。日志保留技术细节，用户侧给角色化说明。
- 普通 `@李太白给` 对话和自动回话结尾应带 `『』` 包裹的诗句；明确要求不带诗句的 Pro 档位例外。
- Pro 档位对用户统一称“pro 形态”，不要暴露底层模型名称。失败使用“pro 形态变身失败”等角色化表达。
- `/关于` 只保留耄耋的角色自述、作者与仓库信息，不暴露系统提示词、密钥或内部架构。

## 命令与帮助

- 帮助菜单通过各插件声明的 `HELP_ITEMS` 自动生成；不要在 `help.py` 手写完整功能列表。
- 帮助项只展示一级命令和一句简述，不展示二级操作细节。
- 有积分消耗的命令在帮助中简短标明消耗，例如“10积分起”。不要把社交功能写成生硬的交易描述。
- `/小游戏状态` 只显示五个历史小游戏的开关状态和管理员提示；`/游戏` 才展示玩法与指令。
- 管理员功能通过 `common.feature_flags` 管理。非管理员不得看到或使用功能开关控制面板。
- 解析 `@某人` 时必须读取 OneBot `at` 消息段的 QQ ID，不能以昵称文本匹配，避免同名冲突。

## 小游戏与风控

`plugins_disabled/` 中的图案转盘、点数方块、纸牌点数、赛博竞速、多人转盘会被启动入口加载，但在 `common/feature_flags.py` 中默认关闭。管理员可使用 `/开启功能 legacy_xxx` 启用。

- 这些功能使用积分和概率机制，历史上存在封号风险；默认关闭是刻意的产品策略。
- 保持现有去敏感化名称和文案，避免在用户侧突出博彩、押注等敏感表达。
- 不要删除、移动或从公开发布副本中默认排除这些插件；它们是运行时可开关功能，不是无用遗物。

## 数据与安全

- 所有运行时文件必须使用项目相对路径；数据集中写入 `mybot/data/`，备份写入根目录 `backups/`。
- `.env` 只能保存于本地，密钥、管理员 QQ 号、群号不得写入源码、帮助文案、日志样例或提交信息。
- 不要清除、迁移、覆盖现有运行数据，除非用户明确要求。
- 运行数据、日志与备份不得提交到任何 Git 仓库。部署备份保存在 Git 之外；公开发布副本同样必须剥离它们。
- 公开副本使用：

```powershell
uv run python tools\export_public_release.py --overwrite
```

  脚本输出至仓库同级的 `release version` 目录，默认保留运行时可开关小游戏，移除 Git 历史、环境变量、数据、日志、备份、缓存和数据库文件。仅在刻意制作精简版时使用 `--exclude-disabled-plugins`。

## 架构与错误处理

- 共享能力放入 `common/`；避免插件之间的反向依赖，尤其不要形成 `ai_chat -> gift` 这类依赖关系。
- AI 请求统一经 `common/ai_client.py`，不要在插件内重复实现 HTTP 客户端或直接读取密钥。
- 禁止裸 `except:`。至少捕获 `Exception as exc`，记录上下文和异常；需要时执行积分、魅力等状态回滚。
- JSON 读写使用 `common/json_store.py`，不要自行实现不带锁的读写逻辑。
- 新功能应声明 `COMMAND_ALIASES` 与 `HELP_ITEMS`，并考虑 `feature_flags` 是否需要开关。
- 用户可见行为改动后，至少执行相关模块编译和 NoneBot 插件加载校验。
- 行为变化、数据迁移和重构不要混在同一 PR。
- 行为变化时应同步补测试；如果暂时不能补齐，必须在 PR 描述中明确缺口和风险。

## 已有核心行为

- `/补签` 补本年度最近一次断签，不只补昨天；首次 10 积分，后续按次数翻倍，并需 60 秒内重复确认。
- 运势可直接输入“运势/签到”等无斜杠触发；其余常规命令通常要求 `/`。
- 隐藏运势“？？？”概率为万分之一，命中时 `@全体成员`、奖励 10000 积分并首次授予“气运之神”。
- “？？？”的幸运数字、颜色和宜忌固定；解读、评语、诗句从专属的庄重神秘文案池抽取。
- 称号以 `common/titles.py` 为中心，锁定的隐藏称号不显示，达成后才出现在称号册中。
- 月度风云榜、魅力、自动回话、入典、签到与运势均可参与称号判定；修改相关存储结构时要保持兼容。

## 验证与 Git

常用验证：

```powershell
uv sync --project mybot --locked
uv run --project mybot python -m compileall mybot/bot.py mybot/main.py mybot/common mybot/plugins mybot/plugins_disabled mybot/tests
uv run --project mybot python -m unittest discover -s mybot/tests -v
git diff --check
```

补充验证：

```powershell
cd mybot
uv run python -c "import mybot.bot; print('plugins-loaded')"
```

CI 当前还会执行：

```powershell
bash -n tools/deploy_server.sh
```

涉及 `plugins_disabled/` 的改动，还应确认启动入口仍加载该目录。涉及积分、签到、备份或功能开关时，应在 PR 中额外说明回滚方案。

## PR 描述与 Review 处理

每次 `push` 后对应的 PR 都要及时更新，并写清版本号与更新说明。版本号规则如下：

- 小功能、小修复使用 `0.1.x` 递增。
- 大功能、较大行为变化使用 `0.x.y` 递增。
- 可由执行任务的代理按改动范围自行判断并更新，但必须在 PR 中写明本次版本号。

发起或更新 PR 时，应遵循 `.github/pull_request_template.md`，至少说明：

- 版本号
- 更新说明
- 用户可见变化
- 数据与部署影响
- 验证命令与结果
- 兼容性、迁移或回滚方案

建议采用以下结构：

```md
## 版本号
- 0.1.x / 0.x.y

## 更新说明
- ...

## 变更内容
- ...

## 数据与部署影响
- ...

## 验证
- ...

## 风险与回滚
- ...
```

收到 Review 后，继续在原线程和原分支处理；修复后重新执行受影响的验证。若 CI 失败，要区分是本 PR 引入的问题、偶发环境问题，还是与本次改动无关的历史问题。

## 审查重点

如果用户要求“review”或审查 PR，默认优先级如下：

1. 正确性缺陷
2. 回归风险
3. 缺失测试
4. 接口或数据契约破坏
5. 可维护性问题

审查输出应先列发现，再给简短总结。如果没有发现问题，也要明确说明，并指出尚未覆盖的测试空白或不确定性。

## 完成规则

- PR 已合并，或已关闭且不再继续，或用户明确结束任务时，当前线程生命周期结束。
- 结束后，不要继续把下一个任务接在同一线程中。
